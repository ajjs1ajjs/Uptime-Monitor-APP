# ⏱️ Параметри моніторингу та таймаути

Повна документація з усіх часових параметрів системи моніторингу.

---

## 📊 Основні параметри

### 1. Інтервал перевірки (Check Interval)

**Параметр:** `check_interval`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `60 секунд` (1 хвилина) |
| **Мінімум** | `30 секунд` |
| **Максимум** | `3600 секунд` (1 година) |
| **Рекомендовано** | `60 секунд` |

**Де налаштувати:**
```json
// /etc/uptime-monitor/config.json
{
  "check_interval": 60
}
```

**Як працює:**
- Кожні 60 секунд система перевіряє всі активні сайти
- Між перевірками окремих сайтів затримка 1 секунда
- SSL сертифікати перевіряються раз на 24 години

---

### 2. Таймаут запиту (Request Timeout)

**Параметр:** `request_timeout_seconds`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `60 секунд` |
| **Мінімум** | `1 секунда` |
| **Максимум** | `300 секунд` (5 хвилин) |
| **Рекомендовано** | `60 секунд` |

**Де налаштувати:**
```json
// /etc/uptime-monitor/config.json
{
  "alert_policy": {
    "request_timeout_seconds": 60
  }
}
```

**Як працює:**
- Якщо сайт не відповідає протягом 60 секунд → помилка "Timeout"
- Статус сайту: `down`
- Відправляється сповіщення (якщо увімкнено)

---

### 3. Поріг помилок для статусу DOWN

**Параметр:** `down_failures_threshold`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `1` (миттєве сповіщення) |
| **Мінімум** | `1` |
| **Максимум** | `10` |
| **Рекомендовано** | `1-3` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "down_failures_threshold": 1
  }
}
```

**Як працює:**
- `1` → сповіщення після першої невдалої перевірки
- `3` → сповіщення після 3 послідовних невдач
- Зменшує хибні спрацювання при тимчасових проблемах

---

### 4. Поріг успіху для статусу UP

**Параметр:** `up_success_threshold`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `1` (миттєве відновлення) |
| **Мінімум** | `1` |
| **Максимум** | `10` |
| **Рекомендовано** | `1-2` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "up_success_threshold": 1
  }
}
```

**Як працює:**
- `1` → статус `up` після першої успішної перевірки
- `3` → потрібно 3 послідовні успіхи для статусу `up`

---

### 5. Повторне сповіщення для STILL DOWN

**Параметр:** `still_down_repeat_seconds`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `600 секунд` (10 хвилин) |
| **Мінімум** | `60 секунд` |
| **Максимум** | `86400 секунд` (24 години) |
| **Рекомендовано** | `600 секунд` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "still_down_repeat_seconds": 600
  }
}
```

**Як працює:**
- Якщо сайт залишається `down`, сповіщення повторюється кожні 10 хвилин
- Уникає спаму сповіщеннями
- Дозволяє відстежувати тривалі простої

---

### 6. Обробка 4xx помилок

**Параметр:** `treat_4xx_as_down`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `true` (вважати помилкою) |
| **Рекомендовано** | `true` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "treat_4xx_as_down": true
  }
}
```

**Як працює:**
- `true` → 404, 403, 401 = `down`
- `false` → 4xx не вважаються критичною помилкою

## 🔒 Параметри SSL моніторингу

### 7. Днів до закінчення сертифікату для сповіщення

**Параметр:** `ssl_notification_days`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `21 день` |
| **Мінімум** | `1 день` |
| **Максимум** | `365 днів` |
| **Рекомендовано** | `21-30 днів` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "ssl_notification_days": 21
  }
}
```

**Як працює:**
- Сповіщення за 21 день до закінчення терміну дії
- Критичне сповіщення за 3 дні (див. `ssl_critical_days`)

---

### 8. Cooltdown між SSL сповіщеннями

**Параметр:** `ssl_notification_cooldown_seconds`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `43200 секунд` (12 годин) |
| **Мінімум** | `300 секунд` (5 хвилин) |
| **Максимум** | `604800 секунд` (7 днів) |
| **Рекомендовано** | `43200 секунд` |

**Де налаштувати:**
```json
{
  "alert_policy": {
    "ssl_notification_cooldown_seconds": 43200
  }
}
```

**Як працює:**
- Уникає повторних сповіщень про той самий сертифікат
- Нове сповіщення не раніше ніж через 12 годин

---

### 9. Критичні дні для SSL

**Параметр:** `ssl_critical_days`

| Значення | Опис |
|----------|------|
| **За замовчуванням** | `3 дні` |
| **Рекомендовано** | `3-7 днів` |

**Де налаштувати:**
```json
{
  "ssl": {
    "ssl_critical_days": 3
  }
}
```

**Як працює:**
- За 3 дні до закінчення → критичне сповіщення
- Термінове оновлення сертифікату

---

## 📈 Статистика часу відповіді

### 10. Час відповіді (Response Time)

**Параметр:** `response_time`

| Метрика | Опис |
|---------|------|
| **Одиниця** | мілісекунди (ms) |
| **Збереження** | Останні 24 години в `status_history` |
| **Агрегація** | AVG, MIN, MAX за 24 години |

**API для отримання:**
```bash
GET /api/response-time/stats
GET /api/response-time/history?site_id=1
```

**Приклад відповіді:**
```json
{
  "site_id": 1,
  "site_name": "Example.com",
  "avg_time": 245.5,
  "min_time": 89.2,
  "max_time": 1234.7,
  "checks": 1440
}
```

## 🔄 Цикл моніторингу

### 11. Перевірка SSL сертифікатів

| Параметр | Значення |
|----------|----------|
| **Інтервал** | Раз на 24 години (86400 секунд) |
| **Час перевірки** | Автоматично (через 25 годин після старту) |
| **Таймаут підключення** | 10 секунд на хост |

**Код:**
```python
# monitoring.py, рядок 410
if (datetime.now() - last_cert_check).total_seconds() >= 86400:
    await check_all_certificates(notify_settings)
```

## 🎯 Рекомендації для різних сценаріїв

### Production (критичні сервіси)

```json
{
  "check_interval": 30,
  "alert_policy": {
    "request_timeout_seconds": 5,
    "down_failures_threshold": 2,
    "up_success_threshold": 2,
    "still_down_repeat_seconds": 300
  }
}
```

**Обґрунтування:**
- Перевірка кожні 30 секунд → швидше виявлення проблем
- Таймаут 5 секунд → швидке виявлення "зависань"
- 2 помилки підряд → уникнення хибних спрацювань
- Повтор кожні 5 хвилин → актуальна інформація

---

### Standard (звичайні сайти)

```json
{
  "check_interval": 60,
  "alert_policy": {
    "request_timeout_seconds": 60,
    "down_failures_threshold": 1,
    "up_success_threshold": 1,
    "still_down_repeat_seconds": 600
  }
}
```

**Обґрунтування:**
- Баланс між швидкістю та навантаженням
- Стандартні налаштування для більшості випадків

---

### Low Priority (некритичні сервіси)

```json
{
  "check_interval": 300,
  "alert_policy": {
    "request_timeout_seconds": 60,
    "down_failures_threshold": 3,
    "up_success_threshold": 2,
    "still_down_repeat_seconds": 1800
  }
}
```

**Обґрунтування:**
- Перевірка кожні 5 хвилин → менше навантаження
- 3 помилки підряд → ігнорування тимчасових проблем
- Повтор кожні 30 хвилин → менше спаму

## 🔧 Команди для перевірки

### Перегляд поточної конфігурації

```bash
sudo cat /etc/uptime-monitor/config.json | jq '.alert_policy'
```

### Перегляд статистики часу відповіді

```bash
# Через API
curl http://localhost:8080/api/response-time/stats

# Через базу даних
sqlite3 /var/lib/uptime-monitor/sites.db "
  SELECT site_id, AVG(response_time) as avg_ms, 
         MIN(response_time) as min_ms, 
         MAX(response_time) as max_ms
  FROM status_history 
  WHERE checked_at >= datetime('now', '-24 hours')
  GROUP BY site_id;
"
```

### Перевірка останніх перевірок

```bash
# Останні 10 перевірок для сайту
sqlite3 /var/lib/uptime-monitor/sites.db "
  SELECT status, status_code, response_time, error_message, checked_at
  FROM status_history 
  WHERE site_id = 1 
  ORDER BY checked_at DESC 
  LIMIT 10;
"
```

## 📞 Вирішення проблем

### Повільний час відповіді

**Проблема:** `response_time > 1000ms`

**Рішення:**
1. Збільште `request_timeout_seconds` до 60
2. Перевірте мережеве з'єднання
3. Оптимізуйте сервер

### Часті хибні спрацювання

**Проблема:** Багато сповіщень про `down`

**Рішення:**
```json
{
  "alert_policy": {
    "down_failures_threshold": 3,
    "up_success_threshold": 2
  }
}
```

### Занадто багато сповіщень

**Проблема:** Спам сповіщеннями

**Рішення:**
```json
{
  "alert_policy": {
    "still_down_repeat_seconds": 1800
  }
}
```
